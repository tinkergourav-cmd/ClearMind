# Global Edit/Preview Mode toggle (v2)

A single `appMode` state atom (`'edit'` | `'preview'`) with a derived `isPreviewMode` flag gates all mutation functions, auto-save effects, context menus, keyboard shortcuts, drag handlers, and the mobile action sheet. Switching to preview cancels in-progress operations and debounced saves; switching back to edit rehydrates from storage. All seven issues from the v1 review (missing declarations, async-inside-updater, ungated undo/redo/z-order) are resolved.

**Watch for:** The state updater still dispatches ~15 `setState` calls as side effects inside `setAppMode`'s updater function — functional in React 18 but non-idiomatic and will double-fire in StrictMode (likely). The "Bring Front" / "Send Back" buttons in the mobile sheet and shift-click node selection remain visually accessible in preview mode even though their backing functions are now guarded (confirmed, minor UX gap).

**Verdict**: APPROVED

## High-level view

The v1 crashes are fixed — `source`, `target`, `task`, and `currentWorkspaces` declarations are all present, and the async hydration now runs outside the updater function body.

The edit→preview cleanup (clearing drag/menu/selection state) still lives inside the `setAppMode` updater rather than alongside it. React 18 batches correctly in production, but StrictMode will double-invoke the updater and fire those 15 nested setters twice. No data corruption, but redundant work and console noise in dev builds.

Two UI elements remain interactive in preview mode without data consequences: the mobile bottom sheet renders "Bring Front" and "Send Back" buttons that silently no-op, and shift-click still toggles node selection with a visible highlight ring but no available action bar.

## State-updater side effects

The edit-to-preview path of `toggleAppMode`:

```javascript
setAppMode(prev => {
  if (prev === 'edit') {
    setEditingTextNode(null);
    setDraggingNode(null);
    draggingNodeRef.current = null;
    setDraggingGroup(null);
    // ... 12 more setState calls ...
    debouncedWorkspaceSaverRef.current.cancel();
    debouncedTaskSaverRef.current.cancel();
    debouncedMetaSaverRef.current.cancel();
    return 'preview';
  } else {
    return 'edit';
  }
});
```

React documents that updater functions should be pure — compute the next state, return it, nothing else. StrictMode deliberately double-invokes updaters to surface impure logic, so each of the 15 nested setters will fire twice. The `.cancel()` calls on the debounced refs also run twice. Since all operations are idempotent (cancelling a cancelled debounce, setting state to `null` twice), no corruption occurs — but future non-idempotent cleanup added here would break silently.

The cleaner pattern: `setAppMode('preview')` as a standalone call, followed by the cleanup setters at the same level. React 18 batches all of them into one render regardless.

## Mobile sheet and selection gaps

In the mobile bottom sheet action grid (line ~7068), "Bring Front" and "Send Back" are rendered unconditionally:

```javascript
{[
  { icon: <ArrowUp />, label: 'Bring Front', action: () => { bringToFront(mobileSheet); ... } },
  { icon: <ArrowDown />, label: 'Send Back', action: () => { sendToBack(mobileSheet); ... } },
  ...(!isPreviewMode ? [
    // Duplicate, Disconnect, Change Theme, Delete
  ] : []),
]}
```

The guard inside `bringToFront` and `sendToBack` prevents state mutation, but the buttons look tappable and silently do nothing — no toast, no disabled styling.

The node `onPointerDown` handler allows shift-click selection before the preview guard:

```javascript
onPointerDown={(e) => {
  // ...
  if (e.shiftKey) {
    setSelectedNodeIds(prev => prev.includes(node.id) ? ... : [...prev, node.id]);
    return;   // <-- returns before isPreviewMode check
  }
  if (isPreviewMode) return;
  // drag logic follows...
```

Selected nodes get a visual highlight ring. The multi-select action bar is hidden (`!isPreviewMode`), so the selection is visible but has no available actions.

## Hydration stale-closure edge case

`toggleAppMode` captures `workspaces` in its closure (used as fallback when hydration returns empty workspaces). Because the callback depends on `[activeProjectId, workspaces, appMode]`, it recreates on every workspace change — but between the last recreation and the point where `await hydrateProject()` resolves, the captured `workspaces` could be stale. This only matters if hydration returns an empty array (triggering the fallback), which is an unlikely runtime path.
