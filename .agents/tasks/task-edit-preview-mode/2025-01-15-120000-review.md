# Edit Mode / Preview Mode toggle for ClearMind

A global `appMode` state (`'edit'` | `'preview'`) replaces the old `editMode` boolean. When preview is active, all three auto-save effects bail early, pending debounced saves are cancelled, editing functions return immediately, drag/resize is blocked, context menus are suppressed, and destructive keyboard shortcuts are gated. Switching back to edit rehydrates the project from storage. Visual indicators (amber header border, canvas ring) signal the current mode.

**Watch for:** Four functions (`duplicateWorkspace`, `cloneNode`, `cloneNodeToWorkspace`, `deleteTask`) lost their local variable declarations during the guard insertion, causing runtime `ReferenceError` crashes when called in edit mode (confirmed). The workspace auto-save effect also references a deleted `currentWorkspaces` variable, crashing on every render cycle in edit mode (confirmed). The toggle's rehydration logic runs an async IIFE inside a React state updater, which is a side-effect-in-reducer anti-pattern that will break under React strict mode double-invocation (confirmed).

## High-level view

The core architecture is the right shape for the feature ‚Äî one state atom, one derived flag (`isPreviewMode`), guards at the operation boundary rather than at individual UI controls. The problem is mechanical: the diff that inserted the guards also deleted variable declarations that still have downstream references, making the app crash in edit mode.

The mode transition has an asymmetry. Entering preview is synchronous and thorough ‚Äî clears all transient state, cancels saves. Returning to edit fires an async hydration inside a `setAppMode` updater function, meaning the state returns `'edit'` immediately while the data reload races behind it. If the hydration fails or is slow, the user is in edit mode with stale data and no error indication.

## Deleted variable declarations (runtime crashes)

The guard insertion pattern `if (isPreviewMode) return;` was added as the first line after `takeSnapshot()` in several functions, but the tool appears to have also removed the `const` declaration that followed. The function immediately references an undeclared variable on the next line.

In `duplicateWorkspace`:
```javascript
const duplicateWorkspace = (wsId) => {
  if (isPreviewMode) return;
  takeSnapshot();
  if (!source) return; // <-- ReferenceError: source is not defined
```

The original had `const source = workspaces.find(w => w.id === wsId);` between `takeSnapshot()` and the null check. The same pattern repeats in `cloneNode` (missing `const target = nodes.find(...)`), `cloneNodeToWorkspace` (same), and `deleteTask` (missing `const task = prev.find(...)` inside the updater callback).

The workspace auto-save effect has the same class of bug: `const currentWorkspaces = workspaces;` was removed but the `for (const ws of currentWorkspaces)` loop on line 1238 still references it. Since this effect runs on every workspace change in edit mode, it will crash immediately on any edit interaction.

## Async side effects inside state updater

`toggleAppMode` places the entire preview-to-edit rehydration inside `setAppMode(prev => { ... })`:

```javascript
setAppMode(prev => {
  // ...
  } else {
    (async () => {
      const hydrated = await hydrateProject(activeProjectId);
      // ... setState calls ...
    })();
    return 'edit';
  }
});
```

React state updaters must be pure functions ‚Äî they should compute and return the next state without side effects. Firing an async IIFE inside one means: (a) the effect runs twice in strict mode dev builds; (b) there's no way to cancel it if the user toggles again before hydration completes; (c) errors inside the IIFE are silently swallowed (no `.catch()`). The hydration should be extracted: set the mode first, then await hydration in the outer `async` function body (the `useCallback` is already marked `async` but never actually awaits anything).

## Gaps in preview enforcement

The mobile bottom sheet still exposes "Bring Front" and "Send Back" actions in preview mode. These call `bringToFront` and `sendToBack` which reorder the nodes array ‚Äî a state mutation that could be captured by a subsequent undo/redo snapshot.

`performUndo` and `performRedo` are not guarded. A user can press Ctrl+Z in preview mode and roll back state, conflicting with the spec's requirement that "no data should be modified."

## Mode transition lifecycle

Returning to edit calls `hydrateProject` which reads from IndexedDB/Firestore. If hydration returns `null` or `undefined`, the existing in-memory state remains ‚Äî but since saves were cancelled in preview mode, that in-memory state may be ahead of what's persisted. There's no user-facing signal that rehydration failed.

The `useCallback` dependency array for `toggleAppMode` captures `workspaces` ‚Äî the callback is recreated on every workspace change. Since `workspaces` is only used as a fallback (`hydrated.workspaces.length > 0 ? ... : workspaces`), this is a stale-closure footgun: the `workspaces` reference captured at callback creation time may be outdated by the time the async hydration resolves. A ref would be more appropriate here.
