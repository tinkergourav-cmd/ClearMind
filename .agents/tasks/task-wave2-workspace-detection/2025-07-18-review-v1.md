# Granular multi-tab detection: project vs workspace collision

The existing multi-tab presence system is extended to distinguish between two collision scenarios: another tab with the same project open (but possibly a different workspace), and another tab with the same project *and* same workspace open. Each tab now broadcasts its `projectId` and `workspaceId` alongside a unique tab ID. A tracking Map of other tabs replaces the previous single-timeout approach, and a `deriveFlags()` function re-evaluates warning state whenever the Map changes. The UI tooltip now shows tiered messages matching the collision type. A localStorage fallback mirrors the same logic for browsers without BroadcastChannel.

**Watch for:** The `useEffect` dependency on `activeProjectId` and `activeTab` tears down and rebuilds the entire presence channel every time the user switches project or workspace, creating a brief window where other tabs see a "leave" followed by a "presence" â€” which may flash the warning off and on in sibling tabs (confirmed). The `otherTabsRef` Map is never cleared on effect re-run, so stale entries from a previous project/workspace context survive into the new effect invocation (confirmed).

## High-level view

The detection model shifts from "is any other tab alive?" to "what project and workspace is each other tab using?" The tracking structure is a per-tab Map keyed by a synthetic tab ID, updated on BroadcastChannel messages and garbage-collected every 5 seconds. Tab crashes are handled via staleness timeout rather than relying solely on `beforeunload`.

The effect depends on `activeProjectId` and `activeTab`, so it tears down and re-establishes the channel whenever the user switches project or workspace. This introduces a transient gap: sibling tabs receive a "leave" from the old effect's cleanup and a "presence" from the new effect's setup, which can briefly flash the warning indicator off then on.

The `otherTabsRef` Map persists across effect re-runs because it's a ref, not local state. After the effect re-mounts with a new project/workspace context, `deriveFlags()` still iterates entries from the *previous* context. These entries will age out in 10 seconds via the cleanup interval, but during that window, stale data drives the flags. Clearing the Map at the top of the effect would eliminate this.

The localStorage fallback carries the same stale-entry problem. Other tabs' entries linger for up to 10 seconds after those tabs switch context, potentially producing false-positive project-level warnings.

## Effect lifecycle and the flash gap

The useEffect at line 1087 depends on `[activeProjectId, activeTab]`. When either value changes, React runs the cleanup function (which posts a "leave" and closes the channel) then runs the effect body fresh (which opens a new channel and posts "presence"). Between those two events, sibling tabs observe the leave, delete the entry from their Map, call `deriveFlags()`, and clear the warning. Milliseconds later the new presence arrives and the warning re-appears.

For a user with two tabs open on the same workspace who switches workspace in one tab, the sibling tab will see: warning removed (leave) â†’ warning re-raised (presence with new identity) or not re-raised (if the new workspace differs). A short debounce (100â€“200ms) on `deriveFlags()` or batching leave+presence into a single "update" message type would eliminate the flash.

## Stale entries in otherTabsRef across re-runs

`otherTabsRef` is a `useRef(new Map())` â€” its `.current` persists for the lifetime of the component, not the lifetime of the effect. When the effect re-mounts after a project switch:

```
// Effect body starts; otherTabsRef.current still has entries from previous run:
// { "tabXYZ" => { projectId: "old-proj", workspaceId: "old-ws", lastSeen: ... } }
```

`deriveFlags()` then compares these entries against the *new* `activeProjectId` and `activeTab` (captured via closure). If `activeProjectId` changed, the old entries won't match and flags stay false â€” benign in that direction. But if only `activeTab` changed while `activeProjectId` stayed the same, stale entries still match on `projectId` and `deriveFlags` sets `isMultiTab = true` based on entries that no longer represent real sibling state (they haven't re-announced with their current context yet).

Adding `otherTabsRef.current.clear()` at the top of the effect body, just after `const myTabId = tabIdRef.current`, solves this. The Map refills within 4 seconds as siblings heartbeat.

## localStorage fallback: stale cross-tab entries

The localStorage fallback doesn't exhibit the Map staleness problem for the *local* tab (since `writePresence()` overwrites the same `myTabId` key with new values). However, `checkOtherTabs()` can still see *other* tabs' stale entries for up to 10 seconds after those tabs switch context. If Tab B switches from Workspace A to Workspace B, Tab A will continue seeing Tab B's old entry (projectId + Workspace A) until Tab B's next heartbeat overwrites it. This produces a false-positive workspace-level warning for up to 4 seconds (the heartbeat interval).
