# Firebase Storage image upload replacing base64 inline storage

The image system moves from encoding dropped images as base64 JPEG strings stored inline in workspace JSON to uploading original files to Firebase Storage and storing download URLs. On drop, an object URL provides immediate visual feedback while the upload proceeds in the background; the URL is swapped to the Firebase download URL on success. Deletion flows (single image, multi-select, workspace, project) all issue corresponding Storage deletes. Backward compatibility is maintained by rendering `img.url || img.src`.

Watch for: the original file is now uploaded at full resolution without compression, which is a regression from the prior 1024px / JPEG 0.7 behavior (confirmed). Object URLs persisted to localStorage/Firestore on upload failure produce broken images on reload (confirmed). Firebase Storage rules are unauthenticated — any client with the config can write/delete arbitrary paths (confirmed).

## High-level view

The drop handler switches from a synchronous base64 pipeline to an optimistic-UI pattern: the image appears immediately via object URL, then gets replaced with the permanent Firebase URL on upload success. When upload fails, the object URL stays in the workspace state — a time-bomb for the next page load since object URLs don't survive beyond the browsing context that created them.

The original code resized images to max 1024px and compressed to JPEG 0.7 before storing. The new code uploads the raw dropped file at full resolution. For photos from modern phones (5-15 MB), this means significantly higher storage costs and slower initial render on subsequent page loads.

Deletion cleanup is wired into all four removal paths. Duplication reuses the existing URL without re-uploading, but this creates a shared-reference problem: deleting the original orphans the duplicate's URL.

No Firebase Auth exists in the codebase, so Storage security rules cannot meaningfully restrict access.

## Raw file upload without resize

The old pipeline created a `<canvas>`, drew the image scaled to max 1024px on either axis, then called `canvas.toDataURL('image/jpeg', 0.7)` — producing a compressed ~50-200 KB payload regardless of source resolution. The new code still calculates display dimensions using the same MAX_SIZE logic but only uses those values for the image element's `width`/`height` attributes. The `file` object passed to `uploadImageToStorage` is the original dropped file, untouched.

For a 12 MP phone photo (~4 MB JPEG, ~15 MB PNG screenshot), this means the Firebase Storage blob is 20-80x larger than the old base64 string would have been. The fix is straightforward: keep the canvas resize, call `canvas.toBlob('image/jpeg', 0.7)`, and pass that blob to `uploadImageToStorage` instead of `file`.

## Object URL lifecycle gap

When upload succeeds, `URL.revokeObjectURL(objectUrl)` is called and the workspace image record gets the permanent Firebase URL. When upload fails:

```javascript
} else {
  console.warn('Image upload failed; keeping local preview for image:', imageId);
}
```

The object URL remains both unrevoked (minor memory leak) and persisted in workspace state. On the next page load — or on another tab that hydrates the same workspace from Firestore — the `blob:http://...` URL is unresolvable and the `<img>` shows a broken image. The workspace JSON in localStorage/Firestore now contains a URL that can never render again.

A resilient approach: on upload failure, either (a) fall back to writing the base64 into the image record as a `src` field (backward-compatible with the old format), or (b) mark the image with a `pendingUpload: true` flag and re-attempt upload on next session load.

## Unauthenticated Storage access

`firebase.js` initializes Storage but there is no Firebase Auth anywhere in the codebase — no `getAuth`, no anonymous sign-in, no auth state listener. Firebase Storage security rules require an auth context to be meaningful. Without auth, the rules must be `allow read, write: if true`, meaning anyone who extracts the Firebase config (it's in the client bundle) can upload arbitrary files, overwrite existing images by guessing the path pattern, or delete other users' images.

## Duplicate creates orphan-prone references

The multi-select duplicate handler copies the image record with a new ID but preserves the existing `url`:

```javascript
const newImages = selectedImages.map(img => ({
  ...img,
  id: `img-${Date.now()}-${Math.random().toString(36).slice(2, 8)}`,
  x: img.x + 40,
  y: img.y + 40,
  workspaceId: activeTab
}));
```

The new image record points to the same Storage path as the original (since the path is `images/{projectId}/{workspaceId}/{originalImageId}`). If the user later deletes the original image, `deleteImage` removes the Storage object, and the duplicate's URL returns 404.

## Deletion coverage and orphan risk

The project-delete path loads workspace data from localStorage to discover image IDs. If localStorage has been cleared (browser storage pressure, manual clear) but Firestore still has the workspace, those Storage files become permanently orphaned. This depends on how reliably localStorage is the source of truth at deletion time. (possible)
