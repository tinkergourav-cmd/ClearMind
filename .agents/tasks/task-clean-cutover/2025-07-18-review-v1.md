# Password Flow Code-Level Review

## Fix Summary

Three (plus one additional) code paths were identified that overwrite existing localStorage password hashes with null/undefined when hydrating project metadata from Firestore (which intentionally strips passwords).

### Fixed Paths

1. **App.jsx initialization hydration loop** (line ~770): Firestore data written to localStorage now preserves existing password hash.
2. **persistenceService.js `initializePersistence()`** (line ~632): Same pattern applied.
3. **App.jsx migration persist block** (line ~873): Additional path discovered during review; same fix applied.

### Path Already Safe

4. **persistenceService.js `hydrateProject()`**: Only writes to localStorage when `loadProjectMeta(projectId)` returns null (i.e., no existing localStorage entry to lose). Added clarifying comment.

## Full Password Flow Review

### changeProjectPassword()

**Classification: CONFIDENT**

- Reads the current project from in-memory `projects` array
- Validates current password via SHA-256 hash comparison
- Hashes new password, updates React state (`setStoredPassword`, `setPasswordEnabled`)
- Updates in-memory projects array
- Reads existing `loadProjectMeta(activeProjectId)` from localStorage, spreads it, adds new password
- Calls `saveProjectMeta` (read-then-write pattern preserves all other fields)
- Calls `saveProjectToFirestore` (which strips password via destructuring -- correct)
- No path can lose the password here

### saveProjectMeta()

**Classification: CONFIDENT**

- Pure write to `localStorage.setItem(KEY_PROJECT_PREFIX + projectId, JSON.stringify(data))`
- The function itself is safe; the risk is in callers passing incomplete data
- All callers that might receive Firestore-sourced data are now patched

### localStorage Persistence (autosave effects)

**Classification: CONFIDENT**

- **Workspace autosave (300ms)**: Reads `loadProjectMeta(activeProjectId)`, spreads it, adds `activeTab`, `workspaceIds`, `lastModified`. Password field preserved because it starts from existing localStorage object.
- **Task autosave (500ms)**: Only writes to `cm-tasks-{projectId}` key. Does not touch project metadata.
- **Metadata autosave (200ms)**: Reads `loadProjectMeta(activeProjectId)`, spreads it, adds `nextId`, `reminders`, `lastModified`. Password field preserved.
- **Password save effect**: Reads `loadProjectMeta(activeProjectId)`, spreads it, explicitly adds `password: storedPassword`. This is the intentional password write path.

### Firestore Hydration (init)

**Classification: CONFIDENT (after fix)**

- App.jsx Firestore init loop: Now reads existing localStorage password before writing
- `initializePersistence()`: Now reads existing localStorage password before writing
- Migration persist block: Now reads existing localStorage password before writing
- Local-is-newer check prevents Firestore overwrite if local data has a more recent `lastModified`

### initializePersistence()

**Classification: CONFIDENT (after fix)**

- Firestore path: iterates all projects from Firestore, hydrates localStorage
- Now preserves existing localStorage password with: `const existingLocal = loadProjectMeta(pid)`
- localStorage fallback path: reads from localStorage directly (already has password)
- Empty state path: no project data exists, nothing to preserve

### hydrateProject()

**Classification: CONFIDENT**

- First tries `loadProjectMeta(projectId)` from localStorage
- If that returns data, it already has the correct password
- Only falls back to Firestore when localStorage has no entry for this project
- In the fallback case, there is no existing password to preserve (localStorage was empty)
- After the project is hydrated, the password flows into the in-memory `projects` array correctly

### Project Switching (switchProject)

**Classification: CONFIDENT**

- Validates target password before switching (hash comparison on in-memory `target.password`)
- Saves current project using read-then-write pattern: `loadProjectMeta(activeProjectId)` -> spread -> `saveProjectMeta`
- Loads target via `hydrateProject(targetId)` which reads from localStorage (with password intact)
- Sets `setStoredPassword(target.password || '')` from in-memory array

### Project Switching (cycleToProject -- passwordless)

**Classification: CONFIDENT**

- Used for default project (no password) and boss key quick-switch
- Same read-then-write pattern for saving current project
- Uses `hydrateProject(targetId)` for loading target
- Sets password state from `target.password` in the in-memory array
- For default project: explicitly clears password (`setStoredPassword('')`)

### Browser Refresh

**Classification: NEEDS MANUAL TEST**

- On refresh, the init useEffect runs
- Firestore path: fixed to preserve localStorage password during hydration
- localStorage fallback: reads from localStorage which has the password
- The `local-is-newer` check may bypass Firestore hydration entirely (preserving local state)
- **Scenario to test**: Set password, close tab, open new tab. Verify password is still required.
- **Scenario to test**: Set password, wait for Firestore sync, refresh. Verify the Firestore hydration does not destroy password.

### Browser Restart

**Classification: NEEDS MANUAL TEST**

- Same as browser refresh but with cold localStorage state
- localStorage is persistent across browser restarts (unless cleared by user/browser policy)
- **Scenario to test**: Set password, fully close browser, reopen. Verify password required.
- **Risk**: If browser is set to clear site data on exit, passwords will be lost (by design -- passwords are localStorage-only)

### Export/Import Behavior

**Classification: CONFIDENT**

- **Export (full backup)**: Exports from in-memory `projects` array which has the correct password hash
- **Export (single project)**: Uses `hydrateProject(targetId)` which reads from localStorage (has password)
- **Import (full backup)**: Writes `password: proj.password || null` from the imported JSON. This is correct because the imported file contains the exported password hash. If password was not in the export, it correctly becomes null.
- The import is a full replacement, not a hydration from Firestore, so this is intentional behavior.

### Duplicate Project Behavior

**Classification: CONFIDENT**

- Explicitly sets `password: ''` on the duplicated project
- This is intentional design: duplicated projects start without password protection
- Saves to localStorage with `password: ''` -- correct
- Does not touch the original project's password

## Remaining Concerns

1. **No remaining paths can silently destroy passwords** after this fix. All Firestore-to-localStorage hydration paths now check for existing password first.

2. **Single point of truth**: Password lives only in `localStorage cm-proj-{id}`. This is by design (credential hashes should not be in Firestore). The trade-off is that passwords do not sync across devices.

3. **No backup mechanism for passwords**: If localStorage is cleared (by user, browser policy, or storage pressure), passwords are permanently lost. This is inherent to the localStorage-only design and documented in the code comments.
