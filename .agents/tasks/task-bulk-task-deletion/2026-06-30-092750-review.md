# Bulk task selection and deletion for the Task panel

The change adds a bulk delete workflow to FullTaskManager that mirrors the existing pattern in PinPanel. Users toggle delete mode via a toolbar button, select tasks with checkboxes, and confirm deletion through a modal. The key behavioral distinction from single-task delete: linked location pins are converted to standalone (pinGroupId set to 'default') rather than being removed.

**Watch for:** Nested `setWorkspaces` call inside the `setTasks` updater creates a state update ordering concern (likely). The `allVisibleTaskIds` useMemo has a dependency on `sortedGroups`, which is a new array reference on every render, making the memo effectively useless (confirmed).

## High-level view

The `bulkDeleteTasks` function in App.jsx nests a `setWorkspaces` call inside a `setTasks` updater function. This is the same pattern used by the existing `deleteTask`, so it's an established convention in this codebase, but it means the workspace state mutation depends on React's internal batching behavior — architecturally fragile for a bulk operation that may touch many pins.

The `allVisibleTaskIds` useMemo lists `sortedGroups` as a dependency, but `sortedGroups` is computed as a new array on every render (`[...groups].sort(...)`). The memo will recalculate on every render regardless.

<details>
<summary>Issues (2)</summary>

1. **Nested state setter in updater** — `bulkDeleteTasks` calls `setWorkspaces` inside the `setTasks` updater callback. If React's batching behavior changes or the component is used in a concurrent context, the workspace update could be dropped or applied out of order. Consider moving the `setWorkspaces` call outside the `setTasks` updater, reading the tasks-to-delete from the current state before calling either setter.
2. **Ineffective useMemo on allVisibleTaskIds** — The dependency array includes `sortedGroups`, which is a new array reference every render. Either memoize `sortedGroups` with `useMemo(() => [...groups].sort(...), [groups])` or switch the dependency to `groups` and `groupedTasks`.

</details>

## Nested state update in bulkDeleteTasks

```javascript
setTasks(prev => {
  const tasksToDelete = prev.filter(t => taskIds.includes(t.id));
  const pinIdsToConvert = tasksToDelete
    .filter(t => t.locationPinId)
    .map(t => t.locationPinId);
  if (pinIdsToConvert.length > 0) {
    setWorkspaces(wsArr => wsArr.map(ws => ({...})));
  }
  return prev.filter(t => !taskIds.includes(t.id));
});
```

Calling `setWorkspaces` from inside the `setTasks` updater is a side effect inside what React expects to be a pure function. React 18 batches these automatically so the user sees a single re-render, but the updater is not supposed to trigger external state mutations. The existing single-task `deleteTask` does the same thing, so this is not a regression — but for a bulk operation touching potentially many tasks, the blast radius of a batching anomaly is larger.

The safer alternative: compute the pin IDs to convert before entering either setter, then call `setWorkspaces` and `setTasks` sequentially at the top level.

## allVisibleTaskIds memoization gap

```javascript
const sortedGroups = [...groups].sort((a, b) => (a.sortOrder || 0) - (b.sortOrder || 0));

const allVisibleTaskIds = useMemo(() => {
  const ids = [];
  sortedGroups.forEach(group => { ... });
  return ids;
}, [sortedGroups, groupedTasks]);
```

`sortedGroups` is declared as a plain `const` computed on every render — each render produces a new array reference. The useMemo dependency check (`Object.is`) will always see a new reference and re-execute the memo. The computation itself is cheap for typical task counts, so the practical impact is negligible, but wrapping `sortedGroups` in its own `useMemo` with `[groups]` as a dependency would fix the intent.
